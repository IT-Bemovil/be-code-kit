## What Is Bemovil?

Bemovil is a Colombian company with 20+ years of operation that started as a mobile minutes/package sales business and evolved into a complex digital products/services platform. It has an integrated payment gateway and a hierarchical business network with commissions. Headquarters in Pereira, Colombia with a secondary office in Buga.

**Purpose**: "Generate wellbeing through technology" — enables even people in precarious conditions to build digital businesses.

## Hierarchy Structure

```
Level 0: Bemovil Root (ID 1 = Colombia, ID 1000 = Ecuador)
  └── Level 1: Clientes Directos (parentId = 1 or 1000)
        └── Level 2+: Clientes Indirectos
              └── Level N: Deep network nodes (NO hardcoded limit)
```

- **No depth limit**: Each distributor can have one or many distributors and/or points of sale beneath them, to arbitrary depth
- **TRI table**: Pre-computed redundancy table that draws every ancestor-descendant relationship. Expensive to recalculate but provides fast hierarchy queries
- **Parent change**: Extremely rare and atypical. Requires full TRI recalculation (costly because TRI is very large). Done directly via database
- **Commission flows UP** the hierarchy — sales at any level benefit all ancestors

## Commission System

Commissions are configured per product per business group:

1. **BusinessGroup** (`business_groups`): Defines a commission group
   - `isDirectToUser`: Whether commissions go directly to the user
   - `topupTip`: Commission on balance top-ups
   - `isAdmin`: Admin-level group flag
   - Multiple businesses can share the same group
2. **Business.businessGroupId**: Links each business to its commission group
3. **BusinessGroupTip** (`business_groups_tips`): Defines the commission for a specific product within a group
   - `tipType`: enum `'percent'` | `'fixed'`
   - `tip`: The commission amount or percentage (double 16,6)
   - `cost`: The cost of the product for this group
   - `productId`: Which product this tip applies to
   - `status`: Status of the tip configuration
4. **Requirement**: A business MUST have a BusinessGroupTip entry for a product (even if value is 0) to sell that product
5. **ProductActivation**: Special products require extra activation by the risk department. Requires additional documentation (camara de comercio, RUT, identity validation, etc.)

```
BusinessGroup ← Business.businessGroupId
     ↓
BusinessGroupTip (per product) → tipType: percent|fixed, tip: N, cost: N
     ↓
Business can sell product IF BusinessGroupTip exists for its group + that product
```

### Commission Scale (Production Data)

- **226,356 commission groups** total
- **13.5M+ tip records** across all groups and products
- **Tip types**: ~7.3M percent-based (avg 1.73%) and ~6.2M fixed-amount (avg $127 COP)
- **Admin groups**: Colombia (id=1) and Ecuador (id=0/1000) with `isAdmin=true`
- **isDirectToUser groups**: Named per distributor, commissions bypass hierarchy and go directly to distributor

- Products table has `options` (JSON) and `providerOptions` (JSON) columns for flexible configuration

- **`opBalance`** (enum): Defines balance operation type

### `providerOptions` JSON Schema (Provider integration config)

The `providerOptions` column defines how the product communicates with external providers:

```jsonc
{
  "query": {                            // Pre-sell query phase (bill lookup, balance check)
    "mapper": [                         // Maps form fields to provider API params
      { "origin": "form", "name": "reference", "payloadSell": true },
      { "origin": "object", "name": "productCode", "payloadSell": false }
    ]
  },
  "sell": {                             // Transaction execution phase
    "mapper": [                         // Maps form fields to sell API params
      { "origin": "form", "name": "phoneNumber", "payloadSell": true },
      { "origin": "form", "name": "value", "payloadSell": true },
      { "origin": "object", "name": "externalId", "payloadSell": true }
    ]
  },
  "wait": {                             // Async polling phase (for providers with delayed response)
    "tries": 10,                        // Max polling attempts
    "delay": 3000,                      // Milliseconds between polls
    "okStatus": "approved",             // Provider response indicating success
    "failStatus": "rejected"            // Provider response indicating failure
  },
  "useBridge": true                     // Whether to route through Sirse SOAP bridge
}
```

**Mapper Pattern**: `origin` can be `"form"` (user input from frontend) or `"object"` (server-side data like product codes). `payloadSell: true` means the field is included in the sell API call to the provider.

## Transaction Engine Patterns

### UUID v7 Primary Key

Transactions use Binary(16) UUID v7 as primary key with custom get/set helpers for efficient storage. The PK is composite: `id` (UUID) + `date` (DATE) — partitioned by date for query performance on the 50M+ row table.

### secureBalance (Pre-deduction Pattern)

Critical anti-double-spending mechanism for async provider calls. Located in `backend/models/transactions/transaction.ts`.

```
1. Transaction created with STATUS_PENDING
2. secureBalance() called: business.balance decremented IMMEDIATELY
3. meta.secured = true, prevBalance/newBalance recorded
4. Provider API called (may take seconds for async providers)
5a. If approved → approve() finalizes, commissions paid
5b. If rejected → reject() detects meta.secured, REVERSES the balance deduction
```

Only applies to TYPE_SALE (3) and TYPE_SALE_MASSIVE (10). Prevents the scenario where two concurrent sales both see sufficient balance and both proceed, overdrawing the account.

### sameReferenceEachMinutes (Anti-fraud Rate Limiter)

Located in `Transaction.sameReferenceEachMinutes()`. Three sequential checks:

1. **No pending transactions**: Rejects if there's a PENDING or PROCESSING transaction for the same reference within the last 3 minutes
2. **Minimum interval**: Rejects if the last transaction on this reference was less than `minTransactionIntervalMinutes` ago
3. **Quantity limit**: Rejects if the total approved transactions for this reference in the configured window exceeds `sameReferenceQuantity`

Configuration comes from product `options.validations`: `sameReferenceEachMinutes`, `sameReferenceQuantity`, `minTransactionIntervalMinutes`.

### Transaction Approval Flow (`approve()`)

Full financial settlement flow in `Transaction.prototype.approve()`:

1. Set status = APPROVED, approvedAt = now
2. For balance loads (opBalance = increment): increment business.balance
3. For sales (opBalance = decrement): if NOT already secured via secureBalance, deduct now
4. **balanceQuota logic**: if business has credit quota (balanceQuota > 0):
   - If balance goes negative → set `nextPaymentQuotaAt` (loan timer starts)
   - If balance returns positive → clear `nextPaymentQuotaAt` + auto-unblock if blocked for MORA
   - If balance drops below `-(2350 + balanceQuota)` → parent business gets auto-blocked
5. Pay commissions via `payCommissions()`
6. Update business.lastTransaction timestamp

### Commission Payment Flow (`payCommissions()`)

1. Walk UP the hierarchy tree (business → parent → grandparent → root)
2. For each ancestor: look up BusinessGroupTip for this product + ancestor's group
3. Calculate commission based on tipType (percent of sale amount, or fixed amount)
4. Create TYPE_SALE_INCREMENT (4) transaction for the ancestor
5. Increment ancestor's profits balance
6. Skip admin IDs {1, 1000} (Bemovil root nodes)

### Rejection Flow (`reject()`)

1. Set status = REJECTED
2. If meta.secured = true → REVERSE the balance deduction (add amount back)
3. Log rejection reason

### Ecuador Sirse Flow (`ecuadorProcessSirse()`)

Ecuador transactions route through Sirse SOAP differently — uses the provider's `className` to dynamically instantiate the correct service class in `ws/class/`. Handles BemovilEcuador provider specifically.

## balanceQuota — Loans/Overdraft System

Businesses can be granted a credit quota (`balanceQuota > 0`) allowing them to go negative up to that amount. This functions as an overdraft/loan:

| Field | Purpose |
|-------|---------|
| `balanceQuota` | Maximum overdraft amount allowed (default 0) |
| `nextPaymentQuotaAt` | When the next loan payment is due (set when balance goes negative) |
| `paymentQuotaEachDays` | Payment frequency in days |

**Flow**:
1. Business sells a product → balance goes negative (within quota limit)
2. System sets `nextPaymentQuotaAt` = now + `paymentQuotaEachDays`
3. If business loads balance and returns positive → `nextPaymentQuotaAt` cleared, auto-unblock if blocked for MORA
4. If balance drops below `-(2350 + balanceQuota)` → parent business gets auto-blocked (cascade protection)
5. Only 186 businesses (0.09%) have balanceQuota > 0 — this is a selective feature

## Sell Flow

1. User selects product → query endpoint validates and returns form inputs
2. User fills form → sell endpoint processes transaction
3. Transaction goes through provider class (Tier 1 direct, or Tier 2 via Sirse SOAP bridge for telecom)
4. Provider responds → transaction recorded → commission calculated up hierarchy
5. Bill/receipt generated → response returned to user

## Key Business Rules

1. **Balance-based sales**: Businesses must have sufficient `balance` to sell. `profits` (commission earnings) can be transferred to `balance`
2. **Commission hierarchy**: Sales generate commissions flowing UP through the entire tree hierarchy, configured via BusinessGroup/BusinessGroupTip
3. **Product activation**: Special products require risk department approval + extra documentation (ProductActivations table)
4. **Multi-country**: Colombia (countryId 5) and Ecuador (countryId 6) with different providers and document types
5. **Identity validation (KYC)**: Progressive identity verification with document types: CC_CO, CC_EC, CC_VE, WORK_PERMIT, PPT
6. **Feature flags**: Products and features can be toggled with configurable conditions via custom feature flag system
7. **Second password**: Some sensitive products require a second password (needSecondPassword field)
8. **MFA**: Optional TOTP-based multi-factor authentication
9. **Device tracking**: Sessions tracked per device with deviceId
10. **Soft deletes**: Most tables use paranoid mode (deletedAt instead of hard delete)
11. **Shared backend**: Both frontend (transactional portal) and admin (internal portal) consume the same backend API
